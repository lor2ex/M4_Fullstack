## 2. Основные подходы к обработке ошибок в Celery

### 2.1. Явное различие типов ошибок

#### Идея

Не все ошибки одинаковы:

* врЕменные или случайные (можно повторить),
* логические (повтор бессмысленен),
* критические (нужно остановиться).

#### Практика

Использование разных исключений:

```python
class TemporaryError(Exception):
    pass

class FatalError(Exception):
    pass
```

И далее:

* `TemporaryError` → retry
* `FatalError` → fail сразу

---

### 2.2. Ретраи как механизм устойчивости

Celery **не должен ретраить всё подряд**.

#### Встроенные ретраи

```python
@app.task(
    autoretry_for=(TemporaryError,),
    retry_kwargs={'max_retries': 5, 'countdown': 30},
    retry_backoff=True,
)
def fetch_data():
    ...
```

Что важно:

* ограничение числа попыток,
* задержка между попытками,
* backoff (увеличение интервала).

Ретраи — это **нормальный путь выполнения**, а не аварийный режим.

---

### 2.3. Явный контроль retry через `self.retry`

Когда нужна логика:

```python
@app.task(bind=True)
def process(self):
    try:
        ...
    except TimeoutError as exc:
        raise self.retry(exc=exc, countdown=60)
```

Плюсы:

* контроль условий,
* разные тайминги,
* учёт состояния.

---

### 2.4. Идемпотентность задач

Простое определение `идемпотентности`: 

**Идемпотентность задачи** означает, что её повторный запуск
* не изменит результат 
* и не навредит системе.

**Примеры идемпотентности из реальной жизни**:

1. **Банковская операция**

   * Перевод 100₽ с одного счёта на другой.
   * Если операция идемпотентна: повторный запрос на перевод не спишет деньги дважды.
   * Если не идемпотентна: деньги могут быть списаны несколько раз.

2. **HTTP-запросы**

   * `GET /user/123` — идемпотентен, т.к. просто возвращает данные, не меняя их.
   * `POST /order` — обычно не идемпотентен, т.к. создаёт новый заказ при каждом вызове.
   * `PUT /user/123` — идемпотентен, т.к. 
     * обновляет данные пользователя до заданного состояния, 
     * и повторное выполнение не изменяет результат.

#### Таким образом,

**Задача должна быть безопасна при повторном выполнении.**

Почему это критично:

* Celery может выполнить задачу повторно,
* воркер может упасть после выполнения, но до ack,
* ретраи неизбежны.

---

#### Очевидные удобства и преимущества идемпотентности

* **Повторная попытка**: 
  * если сервер упал или соединение потерялось, безопасно повторить запрос.
* **Надёжность и отказоустойчивость**: 
  * уменьшается риск дублирования действий.
* **Автоматизация**: 
  * системам проще обрабатывать повторные задачи без ошибок.

---

### 2.5. Dead Letter Queue / обработка «окончательных» ошибок

После исчерпания ретраев задача должна:

* не исчезнуть,
* не крутиться бесконечно.

Подходы:

* отдельная очередь для failed-задач,
* сохранение контекста ошибки в БД,
* ручной разбор и повторный запуск.

---

### 2.6. Таймауты как защита системы

Без таймаутов задача может:

* зависнуть навсегда,
* заблокировать воркер.

В Celery:

* `soft_time_limit` — для аккуратного завершения,
* `time_limit` — жёсткое убийство задачи.

Это **элемент устойчивости**, а не оптимизации.

---

### 2.7. Наблюдаемость (logging, metrics, alerts)

Ошибка, о которой никто не знает, — **хуже, чем ошибка без ретрая**.

Минимум:

* структурированные логи,
* сохранение traceback,
* метрики: success / retry / failure,
* алерты по превышению ошибок.

---

### 2.8. Краткое резюме

| Аспект                 | Зачем                      |
| ---------------------- | -------------------------- |
| Классификация ошибок   | Понимать, что можно лечить |
| Ретраи                 | Переживать временные сбои  |
| Ограничения ретраев    | Не убить систему           |
| Идемпотентность        | Безопасные повторы         |
| DLQ / failure handling | Ничего не терять           |
| Таймауты               | Защита воркеров            |
| Наблюдаемость          | Видеть реальность          |
