## Подготовительный блок: описательный «перевод» ТЗ на человеческий язык**

### 1. Суть проекта

У нас есть два главных процесса:

1. **Основной Процесс (ОП)**
    * это "фабрика новостей", которая работает 24/7 в фоне и **полностью автономно** 
    * управляетсяCelery и Redis (без FastAPI❗) 
    * делает всю тяжёлую работу проекта: 
      * парсит, 
      * фильтрует,
      * генерирует 
      * и публикует.

2. **Интерфейс Настроек Основного Процесса (ИНОП)** 
   * это удобная "человеческая" панель управления (FastAPI + Swagger), 
   * через которую человек (админ/владелец канала) может в любой момент 
     * добавить/убрать источники, 
     * поменять ключевые слова, 
     * посмотреть, что уже опубликовано, 
     * и протестировать генерацию вручную.

Всё взаимодействие этими двумя процессами происходит через Redis:
* по заданным человеком критериям (ключевые слова и источники)
* полностью автоматический процесс
  * временно сохраняет **все** посты из указанных источников,
  * оставляет из них только те, что прошли фильтр
  * и сохраняет их заголовки и url в памяти на сутки, чтобы избежать дублирования


### 2. Схема задач Celery (конвейер обработки)

```
graph LR
    A[Celery Beat<br>каждые 30 мин] --> B[Парсинг источников]
    B --> C[Сырые посты → Redis]
    C --> D[Фильтрация]
    D --> E[Отфильтрованные посты → Redis]
    E --> F[Генерация AI-поста]
    F --> G[Пост готов → Redis]
    G --> H[Публикация в Telegram]
    H --> I[Опубликованные посты → Redis + отметка "опубликовано"]
```

Каждый шаг — отдельная Celery-задача или chain:
- `parse_all_sources()` → `filter_news()` → `generate_post()` → `publish_to_tg()`

Можно запускать цепочку вручную через API для теста любой новости.

### 3. Схема хранения в Redis (что и где лежит)

| Что храним                    | Ключ в Redis (пример)                     | Тип данных          | Назначение |
|-------------------------------|-------------------------------------------|---------------------|----------|
| **Ключевые слова**            | `filters:keywords`                        | Set / List          | Постоянный фильтр |
| **URL + заголовки для дедупликации** | `dedup:seen` (с TTL 24 ч)                | Set (hash: url или title) | Чтобы не публиковать одно и то же дважды |
| **Сырые посты**               | `raw:news:{source_name}:{timestamp}`      | Hash / JSON         | Всё, что только что спарсили |
| **Отфильтрованные посты**     | `filtered:news`                           | List / Sorted Set   | То, что прошло ключевые слова и не дубли |
| **Опубликованные посты**      | `published:posts`                         | List / Sorted Set   | История для API и отчётов |

### 4. Алгоритм обновления данных в Redis (как всё синхронизируется каждые 30 минут)

1. Перед запуском парсинга:
   * Удаляем **все старые сырые посты** кроме самого свежего 
     * (чтобы в API всегда можно было посмотреть «что мы только что собрали» для отладки).

2. Парсим каждый источник:
   * Берём только новости **новее** даты последнего сохранённого поста из этого источника 
   * и + 30 минут назад (чтобы не тянуть древнее).

3. Для каждой новости:
   - Проверяем по ключевым словам → если подходит → кладём в `filtered:news`
   - Проверяем по `dedup:seen` → если уже видели → пропускаем
   - Если прошёл всё → отправляем на генерацию AI

4. После успешной публикации в Telegram:
   - Добавляем пост в `published:posts`
   - Добавляем его url/title в `dedup:seen` (TTL 24 часа)

### 5. Время жизни данных в Redis (TTL)

| Данные                              | TTL                  | Почему именно так |
|-------------------------------------|----------------------|-------------------|
| Сырые посты (кроме самого свежего) | удаляются перед каждым циклом | Чтобы Redis не раздувался, но отладка через API оставалась |
| Отфильтрованные посты               | 1 час (2 цикла)      | Хватает на обработку и повторный запуск цепочки при ошибке |
| Опубликованные посты                | 1 час (2 цикла)      | Достаточно для истории в панели управления |
| Ключевые слова                      | постоянно (без TTL)  | Это настройки, их меняет человек |
| URL + заголовки для дедупликации    | 24 часа              | За сутки новость точно не повторится |

**Итог в одном предложении:**

Мы строим полностью асинхронный конвейер **"«"парсинг → умная фильтрация → AI-журналист → публикация"**,  
где вся промежуточная информация лежит в Redis,  
а человек управляет всем через удобный FastAPI-интерфейс.
