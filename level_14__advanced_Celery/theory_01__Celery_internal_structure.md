Мы уже на практике знаем, что такое Celery.

Теперь давайте разберём Celery «под капотом» максимально глубоко, чтобы наш проект мог бы  
максимально эффективно работать при большой нагрузке.

А для этого ещё раз разберём:
* архитектуру Celery, 
* способы параллелизма, 
* параметры настройки 
* и нюансы возможные при высокой нагрузке. 


---

### 1. Архитектура Celery

Celery — это распределённая система обработки задач (distributed task queue),  
которая позволяет выполнять задачи асинхронно или периодически. 

Основные компоненты:

1. **Broker (посередник сообщений)**

   * Celery **не хранит задачи сам** — он передаёт их брокеру.
   * Популярные брокеры:

     * **RabbitMQ** — надёжный, быстрый, поддерживает подтверждения сообщений и транзакции.
     * **Redis** — проще, быстрый, но не идеально подходит для сложных сценариев доставки сообщений.
   * Задачи помещаются в очередь, которую брокер хранит до момента, пока воркер не заберёт её.

2. **Worker (исполнитель)**

   * Процесс, который забирает задачи из брокера и выполняет их.
   * Можно запускать несколько воркеров на разных серверах или ядрах.
   * Воркер может использовать разные **пулы** (по умолчанию - параллельные процессы):

     * **prefork** — процессы (по умолчанию), каждый воркер отдельный процесс Python. Подходит для CPU-bound задач.
     * **threads** — многопоточность (подходит для I/O-bound задач).
     * **eventlet/gevent** — корутины, асинхронный I/O
       * (по сути — это аналог `asyncio`, с одной существенной разницей:
         * корутиной здесь становится обычный код, без всяких добавок типа `async/await`)
     * **solo** — один поток, для дебага.

3. **Result backend**

   * Celery может хранить результаты выполненных задач.
   * Популярные варианты: Redis, RabbitMQ, PostgreSQL, MySQL, Cassandra, MongoDB.
   * Результат хранится как сериализованное сообщение.

4. **Beat (scheduler)**

   * Для периодических задач, аналог cron.
   * Beat отправляет задачи в брокер по расписанию, и они обрабатываются воркерами.

**Схема работы:**

```
Client (app.delay()) → Broker (RabbitMQ/Redis) → Worker → Result backend
```

### 2. Жизненный цикл задач

1. Создание задачи:

```python
from celery import Celery

app = Celery('myapp', broker='redis://localhost:6379/0')

@app.task
def add(x, y):
    return x + y
```

* `add.delay(2, 3)` — помещает задачу в брокер, не блокируя основной поток.

2. Брокер получает сообщение с сериализованной задачей:

```json
{
  "task": "myapp.add",
  "args": [2,3],
  "kwargs": {},
  "id": "uuid-task-id"
}
```

3. Worker забирает сообщение:

   * Десериализует задачу.
   * Определяет метод и аргументы.
   * Выполняет функцию.
   * Отправляет результат в backend (если настроен).

4. Result backend хранит результат, доступный через `.get()`:

```python
result = add.delay(2, 3)
print(result.get())  # 5
```

### 3. Организация параллелизма

#### 3.1 Пулы воркеров

Celery использует **пул воркеров**, чтобы добиться параллелизма:

| Pool     | Тип параллелизма | Применение                      | Особенности                               |
| -------- | ---------------- | ------------------------------- | ----------------------------------------- |
| prefork  | процессы         | CPU-bound задачи                | Отдельные процессы, GIL не мешает         |
| threads  | потоки           | I/O-bound задачи                | Легче для I/O, GIL блокирует CPU          |
| gevent   | зеленые потоки   | I/O-bound, множество соединений | Асинхронные корутины, monkey patch Python |
| eventlet | зеленые потоки   | I/O-bound, низкая нагрузка      | Требует monkey patching                   |
| solo     | один процесс     | Дебаг / тесты                   | Нет параллелизма                          |

#### 3.2 Конфигурация числа воркеров и concurrency

```bash
celery -A myapp worker --concurrency=4
```

* `concurrency` — число процессов/потоков в пуле.
* По умолчанию `prefork` ставит число равным CPU.

> ⚠️ В данном случае термин `concurrency` выбран КРАЙНЕ неудачно:
> * на самом деле `concurrency` переводится как "квази-параллелизм"
> * но по факту (по умолчанию), создаётся именно реальный параллелизм — 
>   * запускается именно несколько самостоятельных процессов (`parallelism`), 
>   * а не несколько потоков в рамках одного единственного процесса (`concurrency`).

#### 3.3 Prefetching

* Воркеры могут забрать несколько задач сразу (prefetch).
* Настройка через `worker_prefetch_multiplier`.
* Уменьшение значения → задачи распределяются более равномерно, но увеличивается нагрузка на брокер.

#### 3.4 Аcknowledgements

* Celery поддерживает подтверждение выполнения (ack):

  * `ack_late=False` — задача подтверждается при получении (может потеряться при падении воркера).
  * `ack_late=True` — задача подтверждается после выполнения (надёжнее, но может быть повтор выполнений при падении).

### 4. Настройки Celery

Основные группы параметров:

#### 4.1 Брокер

```python
broker_url = 'redis://localhost:6379/0'
broker_transport_options = {
    'visibility_timeout': 3600  # для Redis: время, после которого незавершённая задача возвращается
}
```

#### 4.2 Backend

```python
result_backend = 'redis://localhost:6379/1'
task_serializer = 'json'       # форматы: json, pickle, msgpack
result_serializer = 'json'
accept_content = ['json']      # что принимать
```

#### 4.3 Worker

```python
worker_concurrency = 4         # число процессов/потоков
worker_prefetch_multiplier = 1 # сколько задач за раз воркер забирает
task_acks_late = True          # подтверждение после выполнения
task_time_limit = 300          # max время выполнения задачи
task_soft_time_limit = 200     # предупреждение воркера
```

#### 4.4 Beat

```python
beat_schedule = {
    'add-every-10-seconds': {
        'task': 'myapp.add',
        'schedule': 10.0,
        'args': (16, 16)
    },
}
```

### 5. Параллельность на уровне задач

#### 5.1. `Chain` — последовательное выполнение: `task1.s() | task2.s()`.

```python
from celery import chain

# Два варианта записи
chain(task1.s(), task2.s(), task3.s())()
# или через оператор |
task1.s() | task2.s() | task3.s()
```

* **Что делает:** выполняет задачи **одна за другой**, передавая результат предыдущей задачи в следующую.
* **Особенности:**

  * Полезно, когда **результат первой задачи нужен второй**.
  * Это **последовательная цепочка**, даже если воркеры разные — задачи выполняются по порядку.
* **Пример:**

  * `task1` генерирует данные, `task2` их обрабатывает, `task3` сохраняет результат.

---

#### 5.2. `Group` — параллельное выполнение: `group(task1.s(), task2.s())()`.

```python
from celery import group

group(task1.s(), task2.s(), task3.s())()
```

* **Что делает:** запускает задачи **одновременно** (по возможности, в разных воркерах).
* **Особенности:**

  * Независимые задачи, которые **не зависят друг от друга**.
  * Возвращает объект `GroupResult`, с помощью которого можно отслеживать статус всех задач.
* **Пример:**

  * Скачивание трёх файлов параллельно.
  * Каждая задача делает что-то независимое.

---

#### 5.3. `Chord` — группа + callback после выполнения всех: `chord([task1.s(), task2.s()])(callback.s())`.

*(Читается как "корд")*

```python
from celery import chord

chord([task1.s(), task2.s(), task3.s()])(callback.s())
```

* **Что делает:**

  1. Выполняет **группу задач параллельно** (как `group`).
  2. Когда **все задачи завершены**, запускает **callback**.
* **Особенности:**

  * Полезно для **агрегации результатов** от множества задач.
  * Callback получает **список результатов всех задач группы**.
* **Пример:**

  * Скачиваем 10 файлов, затем объединяем их в один архив.
  * Каждая загрузка независима, а объединение ждёт всех завершений.

---

#### 5.4. `Map/imap` — для массового распределения задач с одинаковыми аргументами.

* **`map`** — синхронно распределяет задачи по аргументам:

```python
from celery import chord

results = chord(task.s(x) for x in range(10))()
```

* **`imap`** — как `map`, но **лениво**, т.е. результаты поступают по мере готовности.
* **Особенности:**

  * Используется, когда нужно **много однотипных задач с разными аргументами**.
  * Позволяет эффективно распределять нагрузку на воркеры.
* **Пример:**

  * Применение функции `process_image(x)` ко всем изображениям в папке.

---

#### 5.5. Ещё раз в одной таблице:

| Механизм       | Параллельность            | Когда использовать                                   |
| -------------- | ------------------------- | ---------------------------------------------------- |
| **Chain**      | Последовательно           | Результат предыдущей задачи нужен следующей          |
| **Group**      | Параллельно               | Независимые задачи, можно сразу запускать все        |
| **Chord**      | Параллельно + callback    | Независимые задачи + финальная агрегация результатов |
| **Map / imap** | Параллельно по аргументам | Много однотипных задач, распределённых на воркеры    |

### 6. Особенности «под капотом»

1. **Serialization**

   * Все задачи сериализуются (`pickle` или `json`) перед отправкой в брокер.
   * Важно учитывать безопасность: pickle опасен с недоверенными данными.

2. **Heartbeat**

   * Воркеры периодически посылают heartbeat брокеру.
   * Если воркер падает — задачи возвращаются (при `ack_late=True`).

3. **Retries**

   * Celery умеет повторять задачи автоматически:

```python
@app.task(bind=True, max_retries=3, default_retry_delay=5)
def unreliable_task(self):
    try:
        do_something()
    except Exception as exc:
        raise self.retry(exc=exc)
```

4. **Event system**

   * Celery воркеры и брокер обмениваются событиями (task-received, task-started, task-succeeded, task-failed).
   * Используется мониторинг (`Flower`, `Celery events`).

5. **Concurrency vs Async**

   * `Prefork` = настоящие процессы → реальный параллелизм (CPU-bound).
   * `Threads/gevent` = кооперативный, подходит только для I/O-bound.
   * `GIL` (Global Interpreter Lock) ограничивает CPU-bound задачи в потоках Python.
