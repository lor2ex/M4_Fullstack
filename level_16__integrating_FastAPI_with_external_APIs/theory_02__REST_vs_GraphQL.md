### 1. REST — определение

**REST (Representational State Transfer)** — это архитектурный стиль взаимодействия клиента и сервера, при котором:

* сервер предоставляет **ресурсы**
* каждый ресурс имеет **URL**
* операции над ресурсами выполняются через **HTTP-методы**
* сервер **не хранит состояние клиента** между запросами (stateless)

**Ключевая идея**:

> *«Я работаю с ресурсами через стандартные HTTP-операции»*

---

### Минимальный пример REST

#### Ресурс: пользователи

```http
GET /users/1
```

Ответ:

```json
{
  "id": 1,
  "name": "Alice",
  "email": "alice@example.com"
}
```

#### Создание пользователя

```http
POST /users
```

```json
{
  "name": "Bob",
  "email": "bob@example.com"
}
```

---

### Что важно в REST

* URL определяет **что** мы запрашиваем
* HTTP-метод определяет **что делаем**
* Ответ имеет **фиксированную структуру**
* Часто возникает:

  * `overfetching` (лишние данные)
  * `underfetching` (не хватает данных → ещё запрос)

---

### 2. GraphQL — определение

**GraphQL** — это язык запросов к API, при котором:

* клиент **сам описывает**, какие данные ему нужны
* запросы идут через **одну точку входа**
* сервер возвращает **ровно ту структуру**, которую запросили

Ключевая идея:

> *«Я описываю форму данных, которые хочу получить»*

---

### Пример GraphQL

#### Запрос

```graphql
query {
  user(id: 1) {
    name
    email
  }
}
```

#### Ответ:

```json
{
  "data": {
    "user": {
      "name": "Alice",
      "email": "alice@example.com"
    }
  }
}
```

**Обратите внимание**:

* нет URL `/users/1`
* нет HTTP-метода для логики
* **структуру определяет клиент**

---

#### Тот же ресурс, но больше данных

```graphql
query {
  user(id: 1) {
    name
    posts {
      title
    }
  }
}
```

#### Ответ:

```json
{
  "data": {
    "user": {
      "name": "Alice",
      "posts": [
        { "title": "Hello World" }
      ]
    }
  }
}
```

**один запрос — ровно то, что нужно**

---

### 3. Ключевое различие на одном примере

#### REST

Получить пользователя и его посты:

```http
GET /users/1
GET /users/1/posts
```

* 2 запроса
* жёсткая структура
* зачастую либо лишние данные, либо их нехватка

---

#### GraphQL

```graphql
query {
  user(id: 1) {
    name
    posts {
      title
    }
  }
}
```

* 1 запрос
* ровно нужные поля
* гибкая структура

---

### 4. Сравнение REST и GraphQL

| Критерий                        | REST                     | GraphQL                                                                       |
| ------------------------------- | ------------------------ |-------------------------------------------------------------------------------|
| Точка входа                     | Много URL                | Один endpoint                                                                 |
| Кто определяет структуру ответа | Сервер                   | Клиент                                                                        |
| HTTP-методы                     | Важны (`GET`, `POST`, …) | Почти не важны                                                                |
| Overfetching                    | Часто                    | Нет                                                                           |
| Underfetching                   | Часто                    | Нет                                                                           |
| Кэширование HTTP                | Простое                  | Сложнее                                                                       |
| Простота                        | Проще                    | Сложнее                                                                       |
| Контроль нагрузки               | Проще                    | Нужно ограничивать, <br>т.к. клиент может запросить <br>СЛИШКОМ много за один раз |

---

### 5. Когда что выбирать

#### REST — если:

* простое CRUD API
* важен HTTP-кэш
* нужен минимальный порог входа
* фиксированные контракты

### GraphQL — если:

* сложные связи данных
* много разных клиентов (web / mobile)
* важно минимизировать количество запросов
* UI часто меняется

---

### 6. Таким образом

* **REST**: *сервер говорит, какие данные ты получишь*
* **GraphQL**: *клиент говорит, какие данные ему нужны*
